# 06 - Notification & Integration Spec: Discord, Telegram, Gmail

## 1. Tujuan Dokumen

Dokumen ini menjelaskan desain sistem integrasi dan notifikasi untuk aplikasi **Project Management**.

Integrasi yang digunakan pada MVP:

1. Discord Webhook.
2. Telegram Bot.
3. Gmail API.

Tujuan integrasi adalah membuat project update tidak hanya tersimpan di aplikasi, tetapi juga sampai ke channel komunikasi yang digunakan tim, founder, CEO, atau client.

---

# 2. Integration Philosophy

## 2.1 Satu Sistem, Banyak Channel

Semua notifikasi harus dikirim melalui satu notification service.

Tujuannya:

1. Logic tidak tersebar.
2. Mudah tracking.
3. Mudah logging.
4. Mudah retry.
5. Mudah menambah provider baru.

---

## 2.2 Channel Harus Punya Peran Berbeda

Jangan semua update dikirim ke semua channel.

### Discord

Untuk update internal tim.

Cocok untuk:

* Task assigned.
* Task completed.
* Project update.
* Daily summary.
* Internal activity.

### Telegram

Untuk alert cepat dan penting.

Cocok untuk:

* Project critical.
* Task urgent overdue.
* Risk analysis high.
* Reminder penting untuk leader.
* Escalation message.

### Gmail

Untuk komunikasi formal.

Cocok untuk:

* Executive summary.
* Weekly report.
* Client update.
* Risk escalation formal.
* Project completion report.

---

```txt

   ↓

   ↓

   ↓

   ↓

   ↓

   ↓
```

# 6. Discord Webhook Integration

## 6.1 Purpose

Discord digunakan sebagai channel update internal tim.

Discord cocok untuk:

1. Update task.
2. Update project.
3. Summary internal.
4. Aktivitas project.
5. Notifikasi ringan yang tidak terlalu urgent.

---

## 6.2 Required Config

```json
{
  "webhookUrl": "https://discord.com/api/webhooks/xxx",
  "channelName": "project-updates"
}
```

Security rules:

1. Webhook URL jangan ditampilkan penuh di UI.
2. Webhook URL disimpan di server/database.
3. Webhook URL tidak dikirim ke client.
4. User hanya melihat masked value.

Masked example:

```txt
https://discord.com/api/webhooks/••••••••••/project-updates
```

---

## 6.3 Save Discord Integration

# 8. Gmail Integration

## 8.1 Purpose

Gmail digunakan untuk komunikasi formal ke CEO, founder, client, atau stakeholder.

Gmail cocok untuk:

1. Weekly executive report.
2. Client progress update.
3. Project risk escalation.
4. Project completion report.
5. Formal status update.

---

## 8.2 Gmail Flow

```txt
User clicks Connect Gmail
   ↓
System generates OAuth URL
   ↓
User authorizes Google account
   ↓
Google redirects to callback
   ↓
Server exchanges code for tokens
   ↓
Tokens stored securely
   ↓
User can send report from Project Management
```

---

## 8.3 Required Config

```json
{
  "connectedEmail": "user@gmail.com",
  "accessToken": "encrypted_access_token",
  "refreshToken": "encrypted_refresh_token",
  "expiresAt": "timestamp"
}
```

Security rules:

1. Token tidak boleh dikirim ke frontend.
2. Token harus disimpan aman.
3. UI hanya menampilkan email yang connected.
4. Jika token expired, server refresh token.
5. User bisa disconnect Gmail.

---

## 8.4 Environment Variables

```env
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
GOOGLE_REDIRECT_URI=
```

---

## 8.5 Gmail Auth URL

Route:

```txt
GET /api/integrations/gmail/auth-url?workspaceId=uuid
```

Response:

```json
{
  "success": true,
  "message": "Gmail auth URL generated successfully",
  "data": {
    "authUrl": "https://accounts.google.com/o/oauth2/v2/auth?..."
  }
}
```

---

## 8.6 Gmail Callback

Route:

```txt
GET /api/integrations/gmail/callback
```

Query:

```txt
code=oauth_code
state=workspaceId
```

Process:

1. Validate state/workspaceId.
2. Validate logged-in user.
3. Exchange code for tokens.
4. Fetch connected email profile if needed.
5. Save token config.
6. Redirect to integration page.

Success redirect:

```txt
/integrations?gmail=connected
```

Failure redirect:

```txt
/integrations?gmail=failed
```

# 9. Notification Service Design

## 9.1 Main Function

```ts
type SendNotificationInput = {
  workspaceId: string;
  projectId?: string;
  provider: "discord" | "telegram" | "gmail";
  eventType: string;
  payload: Record<string, unknown>;
};

async function sendNotification(input: SendNotificationInput) {
  // 1. Create pending log
  // 2. Load integration setting
  // 3. Validate provider enabled
  // 4. Build provider message
  // 5. Send to provider
  // 6. Update log status
}
```

---

## 9.2 Provider Adapter Pattern

Gunakan adapter per provider.

```ts
interface NotificationProvider {
  send(payload: Record<string, unknown>): Promise<{
    success: boolean;
    providerMessageId?: string;
    error?: string;
  }>;
}
```

Adapters:

```txt
DiscordProvider
TelegramProvider
GmailProvider
```

Benefit:

1. Provider logic rapi.
2. Mudah testing.
3. Mudah tambah Slack/WhatsApp nanti.
4. Notification service tidak terlalu besar.

# 14. Security Checklist

```txt
[ ] Discord webhook URL tidak muncul penuh di frontend.
[ ] Telegram bot token tidak muncul penuh di frontend.
[ ] Gmail access token tidak pernah dikirim ke frontend.
[ ] Semua integration route protected.
[ ] Hanya owner/manager yang bisa mengubah integration setting.
[ ] Notification log tidak menyimpan token mentah.
[ ] Error provider tidak membocorkan secret.
[ ] Gmail send wajib melalui preview user.
[ ] Manual notification harus validasi permission.
[ ] Integration config idealnya dienkripsi sebelum disimpan.
```

# 15. Error Handling Standard

## 15.1 Standard Error Response

```json
{
  "success": false,
  "message": "Failed to send notification",
  "error": {
    "code": "TELEGRAM_SEND_FAILED",
    "details": "Telegram API returned an error"
  }
}
```

## 15.2 Frontend Error Display

Tampilkan pesan friendly:

```txt
Telegram alert failed to send. Please check your bot token and chat ID.
```

Jangan tampilkan:

```txt
HTTP 401 raw provider error with full token
```

# 16. Retry Policy

## 16.1 Manual Retry

Untuk MVP, retry cukup manual.

User bisa klik:

```txt
Retry
```

pada notification log yang gagal.

## 16.2 Automatic Retry Future

Untuk production, bisa dibuat automatic retry untuk error sementara.

Retryable:

```txt
network_error
timeout
rate_limited
temporary_provider_error
```

Non-retryable:

```txt
invalid_token
invalid_webhook
missing_permission
invalid_recipient
```

# 18. Integration Success Criteria

Integrasi dianggap berhasil jika:

1. Owner bisa menyimpan Discord webhook.
2. Discord test message berhasil terkirim.
3. Task completed bisa terkirim ke Discord.
4. Owner bisa menyimpan Telegram bot token dan chat ID.
5. Telegram test message berhasil terkirim.
6. AI high risk alert bisa terkirim ke Telegram.
7. User bisa connect Gmail via OAuth.
8. User bisa preview report sebelum kirim email.
9. Gmail bisa mengirim executive report.
10. Semua pengiriman membuat notification log.
11. Semua error integration tampil ramah.
12. Secret tidak bocor ke frontend.
13. Integration settings hanya bisa diubah owner/manager.
14. Demo flow Discord, Telegram, dan Gmail berjalan mulus.
15. CEO bisa melihat bahwa aplikasi ini memiliki automation layer yang nyata.
