# 14 - Service Layer Spec

## 1. Tujuan Dokumen

Dokumen ini menjelaskan standar **service layer** untuk aplikasi **Project Management**.

Service layer adalah tempat utama untuk menyimpan business logic aplikasi.

Tujuan service layer:

1. Mencegah logic menumpuk di `page.tsx`.
2. Mencegah route handler menjadi terlalu besar.
3. Membuat fitur mudah dites.
4. Membuat kode modular.
5. Membuat permission check konsisten.
6. Membuat activity log konsisten.
7. Membuat integration dan AI logic tidak tersebar.
8. Membuat project siap dikembangkan menjadi SaaS yang lebih besar.

---

# 2. Core Principle

## 2.1 Route Handler Harus Tipis

Route handler hanya bertugas:

1. Menerima request.
2. Validasi request body.
3. Mengecek session.
4. Memanggil service.
5. Mengembalikan response.

Route handler tidak boleh berisi business logic panjang.

Contoh buruk:

```txt
app/api/projects/route.ts
```

Berisi:

* Validasi manual.
* Query project.
* Query member.
* Create activity log.
* Calculate progress.
* Send notification.
* Handle error panjang.

Contoh benar:

```txt
app/api/projects/route.ts
```

Hanya memanggil:

```ts
await projectService.createProject(input, context);
```

---

## 2.2 Service Bertanggung Jawab atas Business Logic

Service bertugas:

1. Query database.
2. Validasi akses.
3. Menjalankan business rule.
4. Memanggil service lain jika perlu.
5. Membuat activity log.
6. Menghitung progress.
7. Memanggil AI/integration bila diperlukan.
8. Mengembalikan data bersih ke caller.

---

## 2.3 UI Component Tidak Boleh Langsung Berisi Logic Berat

Komponen UI hanya bertugas:

1. Menampilkan data.
2. Menangani form state.
3. Memanggil action/API.
4. Menampilkan loading/error/success state.

Komponen tidak boleh:

1. Menghitung permission kompleks.
2. Mengambil data dari banyak table secara manual.
3. Memanggil provider eksternal langsung.
4. Menyimpan secret.
5. Menggabungkan logic AI dan notification.

# 4. Service Context Standard

Setiap service sebaiknya menerima `context`.

## 4.1 Service Context Type

```ts
export type ServiceContext = {
  userId: string;
  workspaceId?: string;
  role?: "owner" | "manager" | "member" | "viewer";
};
```

## 4.2 Usage Example

```ts
await projectService.createProject(
  {
    workspaceId,
    name,
    description,
    dueDate
  },
  {
    userId,
    workspaceId,
    role
  }
);
```

## 4.3 Benefit

Dengan context:

1. Permission check lebih mudah.
2. Activity log tahu actor.
3. Service tidak perlu fetch user berulang-ulang.
4. Data access lebih konsisten.

---

# 5. Standard Service Response

Service boleh langsung return data atau menggunakan wrapper.

Recommended wrapper:

```ts
export type ServiceResult<T> = {
  success: boolean;
  message: string;
  data?: T;
  error?: {
    code: string;
    details?: unknown;
  };
};
```

Namun untuk internal service, lebih clean menggunakan:

```ts
return data;
```

Dan error dilempar menggunakan custom error:

```ts
throw new AppError("FORBIDDEN", "You do not have permission.");
```

# 6. App Error Standard

## 6.1 AppError Class

```ts
export class AppError extends Error {
  code: string;
  status: number;
  details?: unknown;

  constructor(code: string, message: string, status = 400, details?: unknown) {
    super(message);
    this.code = code;
    this.status = status;
    this.details = details;
  }
}
```

## 6.2 Common Error Codes

```txt
UNAUTHORIZED
FORBIDDEN
VALIDATION_ERROR
NOT_FOUND
DATABASE_ERROR
AI_GENERATION_FAILED
AI_INVALID_RESPONSE
INTEGRATION_NOT_CONFIGURED
NOTIFICATION_FAILED
GMAIL_AUTH_FAILED
DISCORD_SEND_FAILED
TELEGRAM_SEND_FAILED
```

## 6.3 Error Handling Rule

Service boleh throw error.

Route handler bertugas menangkap error dan mengubahnya menjadi response JSON yang konsisten.

## 11.5 Project Progress Rule

```txt
If total task = 0:
  progress = 0

Else:
  progress = done_tasks / total_tasks * 100
```

Rounded to integer.

---

## 11.6 Project Health Rule

```txt
If progress = 100:
  health_status = completed

Else if due_date < today:
  health_status = critical

Else if due_date <= 3 days from now and progress < 80:
  health_status = at_risk

Else:
  health_status = healthy
```

# 22. Route Handler Pattern

## 22.1 Example Route

```ts
export async function POST(req: Request) {
  try {
    const user = await requireUser();
    const body = await req.json();
    const input = createProjectSchema.parse(body);

    const result = await projectService.createProject(input, {
      userId: user.id,
      workspaceId: input.workspaceId
    });

    return successResponse("Project created successfully", result);
  } catch (error) {
    return errorResponse(error);
  }
}
```

---

## 22.2 Route Handler Rules

1. Jangan query kompleks di route.
2. Jangan simpan business rule di route.
3. Jangan panggil external provider langsung dari route.
4. Jangan membuat response format manual berulang-ulang.
5. Gunakan `successResponse` dan `errorResponse`.

---

# 23. Server Action Pattern

Jika menggunakan server action:

```ts
"use server";

export async function createProjectAction(formData: FormData) {
  const user = await requireUser();

  const input = createProjectSchema.parse({
    name: formData.get("name"),
    description: formData.get("description")
  });

  return await projectService.createProject(input, {
    userId: user.id,
    workspaceId: input.workspaceId
  });
}
```

Rules:

1. Validasi tetap pakai Zod.
2. Permission tetap di service.
3. Error tetap user-friendly.
4. Jangan expose secret.

# 24. Cross-Service Dependency Rule

Service boleh memanggil service lain, tetapi jangan sampai circular dependency.

Recommended dependency direction:

```txt
project-service
  → activity-service
  → notification-service optional

task-service
  → project-service for progress
  → milestone-service for progress
  → activity-service
  → notification-service

ai-service
  → project-service/read data
  → activity-service
  → report-service optional

report-service
  → notification-service
  → activity-service

notification-service
  → integration-service/read settings
  → provider adapters
```

Avoid:

```txt
notification-service → task-service
activity-service → project-service
permission-service → business service
```

---

# 25. Service Layer Success Criteria

Service layer dianggap berhasil jika:

```txt
[ ] Route handler tetap tipis.
[ ] Page component tidak berisi business logic berat.
[ ] Semua permission check konsisten.
[ ] Semua action penting membuat activity log.
[ ] Semua notification attempt membuat notification log.
[ ] AI logic terpisah dari UI.
[ ] Integration provider terpisah dari business service.
[ ] Error handling konsisten.
[ ] Tidak ada secret bocor ke frontend.
[ ] File service tidak terlalu besar.
[ ] Function punya tanggung jawab jelas.
[ ] Kode mudah dites dan dikembangkan.
```

---

# 26. Final Service Layer Rule

```txt
If logic affects business data, put it in service.
If logic validates input, put it in schema.
If logic renders UI, put it in component.
If logic sends external message, put it in provider.
If logic checks access, put it in permission service.
If logic records history, put it in activity service.
```
