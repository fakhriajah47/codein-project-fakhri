# 16 - Route Handler Implementation Map

## 1. Tujuan Dokumen

Dokumen ini adalah peta implementasi route handler/API untuk aplikasi **Project Management**.

Tujuan dokumen ini:

1. Membantu developer/agent tahu route mana yang harus dibuat.
2. Menentukan file path setiap endpoint.
3. Menentukan schema validasi yang dipakai.
4. Menentukan service yang dipanggil.
5. Menentukan permission yang wajib dicek.
6. Menentukan activity log dan notification log yang perlu dibuat.
7. Mencegah route handler menjadi file besar dan berantakan.
8. Menjaga implementasi tetap sesuai PRD.

---

# 2. Route Handler Principle

## 2.1 Route Handler Harus Tipis

Route handler hanya boleh melakukan:

```txt
1. Ambil request.
2. Ambil session/user.
3. Parse body/query params.
4. Validasi dengan Zod.
5. Panggil service.
6. Return success/error response.
```

Route handler tidak boleh:

```txt
1. Berisi business logic panjang.
2. Query banyak table langsung.
3. Menghitung progress manual.
4. Memanggil Gemini langsung.
5. Memanggil Discord/Telegram/Gmail langsung.
6. Membuat permission logic panjang.
7. Menyimpan secret ke response.
```

---

# 3. Standard Route Handler Pattern

```ts
export async function POST(req: Request) {
  try {
    const user = await requireUser();
    const body = await req.json();
    const input = createProjectSchema.parse(body);

    const result = await projectService.createProject(input, {
      userId: user.id,
      workspaceId: input.workspaceId
    });

    return successResponse("Project created successfully", result);
  } catch (error) {
    return errorResponse(error);
  }
}
```

# 4. Standard Response Utility

File:

```txt
lib/utils/response.ts
```

## Success Response

```ts
export function successResponse<T>(message: string, data?: T, status = 200) {
  return Response.json(
    {
      success: true,
      message,
      data
    },
    { status }
  );
}
```

## Error Response

```ts
export function errorResponse(error: unknown) {
  // Convert AppError, ZodError, and unknown error into safe JSON response.
}
```

# 21. Route Implementation Success Criteria

Route implementation dianggap berhasil jika:

```txt
[ ] Semua P0 routes berjalan.

[ ] Semua request body divalidasi Zod.
[ ] Semua query params divalidasi Zod.
[ ] Route handler tetap tipis.
[ ] Business logic berada di service layer.
[ ] Permission check konsisten.
[ ] Activity log tercatat untuk action penting.
[ ] Notification log tercatat untuk pengiriman provider.
[ ] Secret tidak pernah dikirim ke frontend.
[ ] Error response konsisten.
[ ] Success response konsisten.
[ ] API bisa dites manual dengan Postman/Thunder Client.
[ ] API mendukung demo flow dari awal sampai akhir.
```

# 22. Final Route Rule

```txt
If route receives data, validate it.
If route changes data, check permission.
If route triggers business action, call service.
If route sends external message, log it.
If route fails, return friendly error.
If route succeeds, return consistent response.
```
