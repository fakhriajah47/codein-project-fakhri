# 03 - Database ERD & Supabase Schema Design

## 1. Tujuan Dokumen

Dokumen ini menjelaskan desain database untuk aplikasi **Project Management**, yaitu aplikasi project management berbasis AI yang dibangun menggunakan **Supabase Postgres**.

Database harus dirancang agar:

1. Mendukung multi-workspace.
2. Mendukung role-based access.
3. Mendukung project, task, milestone, report, dan integrasi.
4. Aman untuk digunakan dengan Supabase Row Level Security.
5. Mudah dikembangkan untuk fitur lanjutan.
6. Cocok untuk aplikasi portfolio yang terlihat production-ready.

---

# 2. Prinsip Desain Database

## 2.1 Multi-Tenant by Workspace

Semua data utama wajib terhubung ke `workspace_id`.

Tujuannya:

* Data antar workspace tidak tercampur.
* User hanya bisa melihat workspace yang dia ikuti.
* RLS lebih mudah diterapkan.
* Aplikasi bisa berkembang menjadi SaaS multi-user.

Table yang wajib memiliki `workspace_id`:

* projects
* tasks
* activity_logs
* reports
* ai_generations
* integration_settings
* notification_logs

---

## 2.2 User Identity Mengikuti Supabase Auth

Supabase Auth sudah memiliki table internal `auth.users`.

Aplikasi tidak perlu membuat table `users` sendiri. Sebagai gantinya, aplikasi membuat table:

```txt
profiles
```

Table `profiles` digunakan untuk menyimpan informasi tambahan seperti nama lengkap, avatar, job title, dan timezone.

---

## 2.3 Semua Perubahan Penting Harus Tercatat

Setiap perubahan penting harus masuk ke `activity_logs`.

Contoh:

* Project dibuat.
* Task dibuat.
* Task dipindah status.
* Task selesai.
* AI summary dibuat.
* Report dikirim.
* Notification gagal terkirim.

Tujuannya agar aplikasi terlihat profesional dan memiliki audit trail.

---

## 2.4 Data Integration Harus Terpisah dari Business Entity

Data integrasi seperti Discord webhook, Telegram bot token, Gmail OAuth token, dan configuration lain tidak boleh dicampur ke table workspace.

Semua integrasi disimpan di:

```txt
integration_settings
```

Semua hasil pengiriman notifikasi disimpan di:

```txt
notification_logs
```

# 4. Table: profiles

## 4.1 Fungsi

Menyimpan data profile tambahan dari user Supabase Auth.

## 4.2 Columns

```sql
create table profiles (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null unique references auth.users(id) on delete cascade,
  full_name text,
  avatar_url text,
  job_title text,
  timezone text default 'Asia/Jakarta',
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

## 4.3 Notes

* `user_id` harus unique.
* Satu user hanya punya satu profile.
* Profile dibuat otomatis setelah register menggunakan trigger atau logic server.

# 5. Table: workspaces

## 5.1 Fungsi

Workspace adalah container utama untuk semua project, member, role, dan integrasi.

## 5.2 Columns

```sql
create table workspaces (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  slug text not null unique,
  description text,
  logo_url text,
  owner_id uuid not null references auth.users(id) on delete cascade,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

## 5.3 Business Rules

* Setiap workspace wajib punya owner.
* Owner otomatis menjadi member dengan role `owner`.
* Slug harus unique.
* Workspace tidak boleh bisa diakses oleh user yang bukan member.

# 6. Table: workspace_members

## 6.1 Fungsi

Menghubungkan user dengan workspace dan menentukan role user.

## 6.2 Columns

```sql
create table workspace_members (
  id uuid primary key default gen_random_uuid(),
  workspace_id uuid not null references workspaces(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  role text not null check (role in ('owner', 'manager', 'member', 'viewer')),
  status text not null default 'active' check (status in ('invited', 'active', 'suspended')),
  invited_by uuid references auth.users(id) on delete set null,
  joined_at timestamptz,
  created_at timestamptz default now(),
  unique (workspace_id, user_id)
);
```

## 6.3 Role Definition

### owner

Akses penuh terhadap workspace.

### manager

Bisa mengelola project, task, report, dan AI summary.

### member

Bisa mengerjakan task dan berkomentar.

### viewer

Hanya bisa melihat data terbatas.

# 7. Table: projects

## 7.1 Fungsi

Menyimpan data project.

## 7.2 Columns

```sql
create table projects (
  id uuid primary key default gen_random_uuid(),
  workspace_id uuid not null references workspaces(id) on delete cascade,
  name text not null,
  slug text not null,
  description text,
  client_name text,
  project_type text,
  status text not null default 'planning'
    check (status in ('planning', 'active', 'on_hold', 'completed', 'cancelled')),
  priority text not null default 'medium'
    check (priority in ('low', 'medium', 'high', 'urgent')),
  start_date date,
  due_date date,
  progress integer not null default 0 check (progress >= 0 and progress <= 100),
  health_status text not null default 'healthy'
    check (health_status in ('healthy', 'at_risk', 'critical', 'completed')),
  risk_score integer not null default 0 check (risk_score >= 0 and risk_score <= 100),
  created_by uuid not null references auth.users(id) on delete set null,
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  unique (workspace_id, slug)
);
```

## 7.3 Business Rules

* `progress` dihitung dari total task yang selesai.
* Jika semua task selesai, progress menjadi 100.
* Jika due date lewat dan progress belum 100, project masuk status health `at_risk` atau `critical`.
* Project tidak langsung dihapus secara hard delete pada versi production. Untuk MVP boleh hard delete, tetapi best practice-nya pakai archive field.

## 7.4 Recommended Future Column

```sql
archived_at timestamptz
```

# 9. Table: milestones

## 9.1 Fungsi

Menyimpan tahapan besar project.

## 9.2 Columns

```sql
create table milestones (
  id uuid primary key default gen_random_uuid(),
  project_id uuid not null references projects(id) on delete cascade,
  title text not null,
  description text,
  status text not null default 'not_started'
    check (status in ('not_started', 'in_progress', 'completed', 'blocked')),

  progress integer not null default 0 check (progress >= 0 and progress <= 100),
```

## 9.3 Business Rules

# 10. Table: tasks

## 10.1 Fungsi

Menyimpan pekerjaan detail dalam project.

## 10.2 Columns

```sql
create table tasks (
  id uuid primary key default gen_random_uuid(),
  workspace_id uuid not null references workspaces(id) on delete cascade,
  project_id uuid not null references projects(id) on delete cascade,
  milestone_id uuid references milestones(id) on delete set null,
  title text not null,
  description text,
  status text not null default 'todo'
    check (status in ('backlog', 'todo', 'in_progress', 'in_review', 'done', 'blocked')),
  priority text not null default 'medium'
    check (priority in ('low', 'medium', 'high', 'urgent')),
  assignee_id uuid references auth.users(id) on delete set null,
  reporter_id uuid references auth.users(id) on delete set null,
  due_date date,
  estimated_hours numeric(6,2),
  actual_hours numeric(6,2),
  acceptance_criteria jsonb default '[]'::jsonb,
  ai_generated boolean default false,
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  completed_at timestamptz
);
```

## 10.3 Business Rules

* Jika status berubah menjadi `done`, isi `completed_at`.
* Jika status berubah dari `done` ke status lain, kosongkan `completed_at`.
* Task `urgent` dan overdue harus muncul di dashboard alert.
* Task dengan `ai_generated = true` harus tetap bisa diedit user.

# 19. RLS Strategy

## 19.1 Basic Principle

User hanya boleh mengakses data jika dia adalah member dari workspace terkait.

Pseudo rule:

```txt
User can access row if:
exists workspace_members
where workspace_members.workspace_id = row.workspace_id
and workspace_members.user_id = auth.uid()
and workspace_members.status = 'active'
```

---

## 19.2 Helper Function

Disarankan membuat helper function:

```sql
create or replace function is_workspace_member(target_workspace_id uuid)
returns boolean
language sql
security definer
as $$
  select exists (
    select 1
    from workspace_members
    where workspace_id = target_workspace_id
      and user_id = auth.uid()
      and status = 'active'
  );
$$;
```

## 19.3 Helper Function for Role

```sql
create or replace function has_workspace_role(target_workspace_id uuid, allowed_roles text[])
returns boolean
language sql
security definer
as $$
  select exists (
    select 1
    from workspace_members
    where workspace_id = target_workspace_id
      and user_id = auth.uid()
      and status = 'active'
      and role = any(allowed_roles)
  );
$$;
```

# 20. RLS Example: projects

```sql
alter table projects enable row level security;

create policy "workspace members can read projects"
on projects for select
using (is_workspace_member(workspace_id));

create policy "owner and manager can insert projects"
on projects for insert
with check (has_workspace_role(workspace_id, array['owner', 'manager']));

create policy "owner and manager can update projects"
on projects for update
using (has_workspace_role(workspace_id, array['owner', 'manager']));

create policy "owner can delete projects"
on projects for delete
using (has_workspace_role(workspace_id, array['owner']));
```

# 21. RLS Example: tasks

```sql
alter table tasks enable row level security;

create policy "workspace members can read tasks"
on tasks for select
using (is_workspace_member(workspace_id));

create policy "owner manager member can insert tasks"
on tasks for insert
with check (has_workspace_role(workspace_id, array['owner', 'manager', 'member']));

```

# 22. Progress Calculation Rules

## 22.1 Project Progress

Formula:

```txt
project_progress = done_tasks / total_tasks * 100
```

Jika project tidak punya task:

```txt
progress = 0
```

## 22.2 Milestone Progress

Formula:

```txt
milestone_progress = done_tasks_in_milestone / total_tasks_in_milestone * 100
```

## 22.3 Health Status Rule

```txt
if progress = 100:
  health_status = completed

else if due_date < today and progress < 100:
  health_status = critical

else if due_date is within 3 days and progress < 80:
  health_status = at_risk

else:
  health_status = healthy
```

## 22.4 Risk Score Basic Formula

Sebelum AI analysis, sistem bisa menghitung risk score awal:

```txt
risk_score =
  overdue_task_ratio * 40
  + urgent_open_task_ratio * 25
  + blocked_task_ratio * 20
  + deadline_pressure_score * 15
```

AI kemudian membantu menjelaskan alasan dan rekomendasi.

# 24. Database Success Criteria

Database dianggap berhasil jika:

1. User bisa register dan memiliki profile.
2. User bisa membuat workspace.
3. User otomatis menjadi owner workspace.
4. Workspace bisa memiliki banyak member.
5. Project hanya terlihat oleh member workspace.
6. Task hanya terlihat oleh member workspace.
7. Project progress berubah ketika task selesai.
8. Activity log tercatat setiap ada perubahan penting.
9. AI generation tersimpan.
10. Report tersimpan.
11. Notification log tersimpan.
12. Integration settings tersimpan per workspace.
13. RLS mencegah akses data lintas workspace.
14. Query dashboard berjalan cepat.
15. Schema mudah dikembangkan untuk fitur lanjutan.
